# 五間公司帳務管理優化方案

> 工作台參考文件 — 2026-03-17

---

## 一、LINE 官方帳號數量限制

**結論：可以。一個 LINE 帳號可以建立多個官方帳號（LINE Official Account），沒有數量上限。**

- 一個管理者 Email 可以同時管理多個官方帳號
- LINE 官方帳號管理後台支援一鍵切換，不需重複登入
- 每個官方帳號獨立擁有 Channel ID / Channel Secret / Webhook
- 每個官方帳號可以各自設定管理員權限，讓不同員工管理不同公司

### 費用考量

| 方案 | 月費 | 免費訊息 | 加購訊息 |
|------|------|---------|---------|
| 免費 | $0 | 200 則 | 不可加購 |
| 輕用量 | $800 | 3,000 則 | $0.2/則 |
| 中用量 | $4,000 | 25,000 則 | $0.15/則 |

5 個官方帳號若都用免費方案 = $0/月（各 200 則訊息）
5 個輕用量 = $4,000/月

## 二、三種架構方案比較

### 方案 A：五個獨立 LINE 官方帳號（推薦度 ★★★☆☆）

```
客戶 LINE → [公司A官方帳號] → shanbot-A (port 8025)
客戶 LINE → [公司B官方帳號] → shanbot-B (port 8026)
客戶 LINE → [公司C官方帳號] → shanbot-C (port 8027)
...
```

**優點：**
- 帳務完全隔離，零串帳風險
- 每間公司獨立報表、獨立 GDrive 資料夾
- 稅務匯出簡單（每個 bot 獨立一套）

**缺點：**
- 需部署 5 個 shanbot 實例（RAM 吃緊）
- 客戶需加入 5 個 LINE 好友
- 維護成本高（更新要改 5 份程式碼）
- 月費可能較高

---

### 方案 B：單一 BOT + 公司選擇器（推薦度 ★★★★★）

```
客戶 LINE → [小膳官方帳號] → shanbot (port 8025)
                                  ├── 公司A 帳本
                                  ├── 公司B 帳本
                                  ├── 公司C 帳本
                                  ├── 公司D 帳本
                                  └── 公司E 帳本
```

**流程設計：**
1. 客戶上傳收據照片
2. OCR 辨識後，彈出「公司選擇器」Flex Message：
   ```
   ┌─────────────────────────────────┐
   │  📋 請選擇這張收據屬於哪間公司  │
   ├─────────────────────────────────┤
   │  🏢 A 餐飲有限公司             │
   │  🏢 B 團膳事業                 │
   │  🏢 C 食品行                   │
   │  🏢 D 餐盒工廠                 │
   │  🏢 E 中央廚房                 │
   └─────────────────────────────────┘
   ```
3. 客戶點選後，系統自動歸入該公司帳本
4. 所有報表、GDrive 資料夾依公司分開

**資料庫變更：**
- `purchase_staging` 加 `company_id` 欄位
- 新增 `companies` 資料表（id, name, tax_id, address, ...）
- 報表/匯出/GDrive 全部按公司分隔

**GDrive 資料夾結構：**
```
小魚資料/
├── A餐飲有限公司/
│   ├── 2026/03月/收據憑證/...
│   └── 2026/03月/待確認/...
├── B團膳事業/
│   ├── 2026/03月/收據憑證/...
│   └── ...
└── ...
```

**優點：**
- 只需一個 LINE 官方帳號、一個 shanbot 實例
- 客戶操作最簡單（一個 LINE 好友就搞定）
- 維護成本最低
- 帳務按公司完全分開，報表獨立
- 可以做跨公司彙總分析

**缺點：**
- 需要開發公司選擇器功能
- 第一次使用需要設定公司清單

---

### 方案 C：單一 BOT + 智慧分類（推薦度 ★★★★☆）

和方案 B 類似，但加入 AI 自動判斷公司歸屬：

1. OCR 辨識收據後，根據以下線索自動推斷公司：
   - **統一編號**：每間公司統編不同，發票上的「買方統編」直接判斷
   - **供應商綁定**：某些供應商只供貨給特定公司
   - **歷史記錄**：同一供應商過去歸屬哪間公司
2. 高信心度自動歸類，低信心度才彈出選擇器
3. 客戶可以隨時修改歸屬

**優點：**
- 多數情況下客戶不需要手動選擇
- 學習效果越用越準

**缺點：**
- 初期需要較多手動選擇建立學習基礎
- 買方統編辨識需要額外開發

## 三、建議實施方案

**推薦：方案 B（公司選擇器）→ 逐步進化為方案 C**

### Phase 1：公司選擇器（2-3 天開發）

1. 新增 `companies` 資料表
2. `purchase_staging` 加 `company_id` 欄位
3. OCR 後彈出公司選擇 Flex Message（Quick Reply 或 Postback）
4. GDrive 資料夾按公司名稱建立子目錄
5. 報表生成加上公司篩選

### Phase 2：智慧分類（+1-2 天）

1. 供應商→公司綁定表
2. 發票買方統編自動匹配
3. 歷史記錄推測

### Phase 3：跨公司分析（+1 天）

1. 五公司費用比較表
2. 合併損益報表
3. 統一供應商管理（跨公司議價）

## 四、需要 Simon 確認的事項

1. **五間公司名稱和統一編號**（用於建立 companies 資料表）
2. **偏好方案**：A（五個 BOT）或 B（單一 BOT + 選擇器）？
3. **GDrive 結構**：按公司分頂層目錄，還是按年月分頂層、公司分子層？
4. **報表需求**：需要跨公司合併報表嗎？
5. **月費預算**：如果選方案 A，五個輕用量月費 $4,000 可接受嗎？
